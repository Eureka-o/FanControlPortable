# FanControl 2.2.0 设备与设备功能迁入计划

日期：2026-06-07

## 2026-06-07 范围修订：飞智暂不进入前端设备库

当前执行边界改为：
- 飞智（FlyDigi）BS1/BS2/BS2PRO/BS3/BS3PRO 暂不作为用户可见设备保存到 FanControl 设备库，也不在设备管理、支持设备列表或普通设置页中展示。
- 前端可见设备连接与设备管理先保持 2.1.2 的正式发行体验；飞智相关搜索卡、自动识别入口和设备功能入口不开放给普通用户。
- 后端飞智协议、HID/BLE 枚举、命令和解析代码可以保留作为 Beta/后续验证基础，但不得自动写入 `deviceProfiles`、`activeDeviceProfileId` 或 `activeDeviceProfileIdsByTransport`。
- 配置规范化时应清理本轮开发中误写入的飞智隐藏档案，避免影响 2.0/2.1 用户升级继承。
- 2.2.0 当前重点调整为 WiFi/虚拟串口兼容模式、WiFi IP 扫描/重连基础、设备档案解耦和配置继承；飞智前端设备库与原生功能回归需等真实验证和再次确认后再开放。

下文关于“飞智作为 5 个内置设备进入前端设备库、自动启用并显示设备功能”的内容保留为历史草案，不作为当前执行要求。

本文档记录 FanControl 2.2.0 第一阶段计划：先把参考软件中已支持的飞智设备作为独立设备档案迁入 FanControl，把参考软件原本支持的飞智原生功能接回，并把设备功能改成白名单/能力驱动。本文档只描述计划，不代表已经完成实现。

## 目标

- 把参考软件中的 5 个飞智设备作为 5 个独立内置设备加入 FanControl 设备库。
- 所有型号名前加品牌前缀：`飞智（FlyDigi）`。
- 设备功能按设备档案白名单显示和允许调用，不按连接方式粗暴推断。
- 设备管理界面能在设备名称下方显示功能小胶囊。
- 设置页能根据当前启用设备显示不同的设备功能区域。
- WiFi 连接增加 IP 扫描/发现能力，帮助用户在同网段或电脑热点场景下找到当前设备地址。
- 2.2.0 只恢复飞智等参考软件原本支持设备的原生智能启停；WiFi 软件模拟智能启停推迟到 2.2.1。
- 保留 2.0/2.1 配置、曲线、主题、IP、设备档案、学习数据兼容。
- 不影响原作者 THRM/reference 版本的独立运行。

## 第一阶段范围

第一阶段只做“设备和设备功能搬进来”的基础层，不急着一次性完成所有真实设备协议验证。

包含：

- 内置设备档案。
- 协议模板标识。
- 功能能力白名单。
- 设备管理功能胶囊。
- 设置页设备功能分区。
- 飞智 HID 设备原生功能接回：智能启停、通电自启、挡位灯、灯光/RGB、有限亮度等参考软件已有能力。
- WiFi IP 扫描入口和后端扫描能力。
- 后端能力校验。
- 兼容和回归测试。

暂不强行包含：

- 新增未经验证的 WiFi 灯光/通电自启/原生智能启停接口。
- WiFi 软件模拟智能启停、停机值探测、恢复状态机和温度保护；这些放到 2.2.1。
- 小屏幕功能。
- 对所有 HID/BLE 设备做真机验证。
- 自动发布 release。

## 飞智设备自动匹配

飞智（FlyDigi）设备应优先自动匹配，不要求普通用户手动选择型号。

自动匹配规则：

- HID 设备通过 VID/PID 匹配：
  - VID `0x37D7`。
  - BS2 PID `0x1001`。
  - BS2PRO PID `0x1002`。
  - BS3 PID `0x1003`。
  - BS3PRO PID `0x1004`。
- BLE BS1 通过 BLE 扫描信息匹配：
  - 广播名包含 `BS1` 或参考软件已知的 FlyDigi BS1 名称。
  - 服务/特征优先匹配 `FFF0` / `FFF2` / `FFF1`。
- 只发现一台匹配设备时，自动选择对应内置设备档案。
- 发现多台同类设备或识别结果冲突时，弹出设备选择，不静默乱连。
- 自动匹配失败时，保留设备管理里的手动选择/高级调试入口。

UI 行为：

- 连接成功后，主页和设备管理显示自动识别出的设备名，例如 `飞智（FlyDigi）BS2PRO`。
- 设置页根据识别出的设备档案自动显示对应设备功能。
- 设备管理中仍允许用户查看、启用或禁用设备档案，但普通连接流程不要求先手动选飞智型号。

## 协议模板

第一阶段至少建立这些协议模板概念：

| 模板 | 用途 |
|---|---|
| `flydigi-frame-v1` | 飞智共享帧：`5A A5 cmd len payload checksum` |
| `flydigi-hid-rpm-v1` | BS2/BS2PRO/BS3/BS3PRO 共用 HID RPM 协议，HID ReportID `0x02` |
| `flydigi-bs1-ble-rpm-v1` | BS1 BLE GATT RPM 协议，服务 `FFF0`，写 `FFF2`，通知 `FFF1` |
| `fancontrol-wifi-percent-v1` | 当前 FanControl WiFi 百分比控制协议 |

实现要求：

- 共享帧编码/解码不要复制到每个型号里。
- HID 和 BLE 传输层分开。
- 百分比 WiFi 控制路径继续独立于飞智 RPM 路径。
- 设备档案引用协议模板，UI 不直接判断型号来决定发包。

## 功能白名单

默认原则：

- 未确认的设备功能一律不显示、不允许调用。
- 前端隐藏只是体验层；后端必须再次检查能力。
- 连接方式不等于设备能力。BLE/HID/WiFi/串口都不能自动推出灯光、小屏幕、智能启停等功能。

计划能力表：

| 设备 | 风速控制 | 状态读取 | 手动挡位 | 自定义转速/百分比 | 通电自启 | 智能启停 | 挡位灯 | 灯光/RGB | 亮度 | 小屏幕 |
|---|---|---|---|---|---|---|---|---|---|---|
| Slim压风散热器Pro | 是 | 是 | 按现有映射 | 是 | 否 | 否 | 否 | 否 | 否 | 否 |
| 飞智（FlyDigi）BS1 | 是 | 是 | 4 个固定挡位 | 是 | 是 | 否 | 否 | 否 | 否 | 否 |
| 飞智（FlyDigi）BS2 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 有限 | 否 |
| 飞智（FlyDigi）BS2PRO | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 是 | 有限 | 否 |
| 飞智（FlyDigi）BS3 | 是 | 是 | 是 | 是 | 是 | 是，参考兼容 | 是，参考兼容 | 是，参考兼容 | 有限，参考兼容 | 否 |
| 飞智（FlyDigi）BS3PRO | 是 | 是 | 是 | 是 | 是 | 是，参考兼容 | 是，参考兼容 | 是，参考兼容 | 有限，参考兼容 | 否 |

说明：

- BS1 在参考运行时明确不支持智能启停、挡位灯、灯带/RGB、亮度。
- HID 四个型号走参考软件非 BS1 共用路径，但 BS3/BS3PRO 仍建议标注为参考兼容、待实机确认。
- 亮度是“有限”能力，因为参考单独亮度接口主要处理关闭/全亮；灯效里的亮度参数另算。
- 小屏幕功能五个飞智设备先全部关闭，后续有协议或真机反馈再加入白名单。

## 设置页设备功能

设置页计划拆成设备相关的二级区域：

```text
设置
├─ 连接
├─ 设备功能
│  ├─ 通电自启
│  ├─ 智能启停
│  ├─ 挡位灯
│  ├─ 灯光 / RGB / 灯带
│  └─ 未来小屏幕 / 其它硬件功能
├─ 高级调试
└─ 系统
```

规则：

- 灯光属于设备功能。
- 灯光内容较多时，可以在“设备功能”下单独折叠为“灯光”分区。
- 当前设备没有任何可配置设备功能时，隐藏设备功能区域或显示简短空状态。
- BS1 不显示灯光和智能启停。
- Slim压风散热器Pro 不显示灯光、通电自启、智能启停。
- HID 飞智设备按白名单显示对应开关和选项。

## 智能启停菜单位置与方案

智能启停属于设备功能，不属于连接设置。2.2.0 只接回参考软件原本支持设备的固件原生智能启停，主要是飞智 HID 设备；WiFi 软件模拟智能启停推迟到 2.2.1。日常使用入口放在设置页的设备功能里；方案参数和高级策略放在高级设备管理的设备档案里。

菜单位置：

```text
设置
└─ 设备功能
   └─ 智能启停 [方案胶囊]
      ├─ 开关
      ├─ 延时/停止时机
      └─ 简短说明
```

显示规则：

- 设备功能页负责显示智能启停入口。
- 连接页只负责扫描、启用设备、兼容模式和必要连接参数。
- 高级设备管理负责智能启停方案设置、设备档案、能力声明、协议模板和高级调试。
- 如果当前设备没有任何可用智能启停方案，则隐藏入口，或在设备功能空状态里提示“当前设备不支持智能启停”。
- 2.2.0 中，如果当前设备有固件原生方案，则显示 `智能启停` 入口。
- 软件模拟方案、停机值探测、恢复状态机和温度保护属于 2.2.1，不在 2.2.0 普通设置页显示。
- 设置页里的智能启停只做日常使用，不直接编辑停机值、延迟、阈值、回滚策略、协议命令等高级参数。
- 智能启停方案显示在标题旁边，例如 `智能启停  原生`；`软件模拟·实验` 只作为 2.2.1 预留。
- 普通设置页交互应接近参考软件的轻量体验：一个开关控制启用/关闭，一个选项控制延时或停止时机，再加一段简短说明。

智能启停方案建议：

| 方案 | 显示名 | 适用设备 | 说明 |
|---|---|---|---|
| `native` | 原生智能启停 | 飞智 HID 等固件支持设备 | 设备自己执行启停逻辑，FanControl 只负责下发配置 |
| `software` | 软件模拟智能启停 | 2.2.1 起评估 WiFi/串口等仅支持调速的设备 | FanControl 通过 `0%` 或停机值模拟，需要探测、回滚和温度保护 |
| `shutdown_send` | 退出/关机前停机 | 2.2.1 起评估不支持后台延时但支持停机值的设备 | FanControl 在退出或关机前尽量发送停机/恢复命令，不保证关机后延时执行 |
| `disabled` | 关闭 | 全部设备 | 不启用智能启停 |

不同设备的入口表现：

- 飞智（FlyDigi）BS1：不显示智能启停入口，除非后续有真机证据证明支持。
- 飞智（FlyDigi）BS2/BS2PRO/BS3/BS3PRO：显示智能启停，默认方案为 `原生智能启停`。
- Slim压风散热器Pro：2.2.0 不显示智能启停入口；WiFi 软件模拟方案推迟到 2.2.1。
- DIY WiFi/串口设备：2.2.0 只在设备档案明确声明固件原生命令时显示原生方案；停机值/最低安全值的软件模拟方案推迟到 2.2.1。

UI 要求：

- 智能启停标题旁用小胶囊标明方案，例如 `原生`、`软件模拟`、`实验`、`需要测试`。
- 2.2.0 说明文字只写用户需要知道的结论：当前设备由固件执行。软件模拟说明在 2.2.1 再加入。
- 普通设置页不要放调试日志、停机值、恢复值、安全阈值、失败次数等高级字段。
- 不要把 WiFi 兼容模式开关和智能启停方案混在一起；兼容模式只是连接方式，智能启停是设备功能。

高级设备管理中的方案设置：

- 原生方案：命令模板、模式映射、状态解析、支持的模式。
- 软件模拟方案：2.2.1 再加入停机值、最低安全值、恢复速度、启动阈值、停止阈值、温度迟滞、最小运行时间、最小停机时间、失败次数上限。
- 退出/关机前停机方案：2.2.1 再加入退出时动作、关机前动作、失败后的恢复策略。
- 测试记录：最近一次停机测试、恢复测试、失败原因和用户确认状态。
- 这些高级参数跟随设备档案导入/导出，不放在普通设置页里直接编辑。

添加/编辑设备中的智能启停字段：

- 添加设备或编辑设备时，可以配置“支持智能启停”开关。
- 2.2.0 开启后只选择 `原生智能启停`，用于参考软件已支持的固件命令设备。
- `软件模拟智能启停`、`退出/关机前停机` 和对应高级字段推迟到 2.2.1。
- 这些设置只影响设备档案能力和方案，不直接出现在普通设置页里。

## 设置页连接逻辑

当前设置页的连接操作需要调整：设备管理负责“设备库/模板/高级协议”，设置页负责“日常启用设备和连接方式”。普通用户不应该为了连接飞智 BLE/HID 设备先去设备管理手动选择型号。

建议结构：

```text
设备连接
├─ 自动扫描设备
│  ├─ 扫描蓝牙 BLE
│  ├─ 扫描 HID
│  └─ 显示自动匹配结果
├─ 当前启用设备
├─ 兼容模式
│  ├─ 启用 WiFi 兼容模式
│  │  ├─ 启用设备
│  │  └─ IP 地址
│  └─ 启用虚拟串口/COM 兼容模式
│     ├─ 启用设备
│     ├─ 端口
│     └─ 波特率
└─ 高级设备信息入口
```

连接方式规则：

- 日常入口应优先是“自动扫描设备”按钮，而不是让用户先手动选择 BLE/HID 型号。
- 主页刷新本质上也应触发一次轻量扫描/状态刷新：重新枚举 BLE/HID 自动匹配设备，并刷新当前启用设备状态。
- “当前启用设备”应放在设置页日常连接区域里，而不只放在设备管理页面。
- 设备管理继续用于新增设备、导入/导出设备、编辑设备档案、维护高级协议和调试。
- BLE 和 HID 对内置/已录入且具备匹配规则的设备应默认自动扫描、自动匹配、自动套用设备档案，不作为常规手动配置项长期占用设置页。
- WiFi 和虚拟串口/COM 作为“兼容模式”出现，默认关闭或不强调，只有开启后才允许使用对应连接方式。
- WiFi 兼容模式开启后，应先显示“启用设备”选择，再在下方显示 IP 地址小栏。
- 虚拟串口/COM 兼容模式开启后，应先显示“启用设备”选择，再在下方显示端口和波特率小栏。
- 兼容模式旁边必须有说明：WiFi/虚拟串口用于 DIY 主控、旧设备或无法通过 BLE/HID 自动识别的设备，需要手动配置连接参数。
- BLE/HID 自动匹配失败、发现多台同类设备、或匹配信息冲突时，再显示手动选择。

推荐 UI 行为：

- 用户点击 `自动扫描设备`：同时扫描/枚举 BLE 与 HID，匹配飞智（FlyDigi）BS1、BS2、BS2PRO、BS3、BS3PRO 或其他已录入自动匹配设备。
- 扫描只找到一台匹配设备：自动启用并刷新主页、设置页、设备管理显示名称。
- 扫描发现多台匹配设备：显示选择器，让用户确认启用哪一台。
- 用户开启 `WiFi 兼容模式`：显示启用设备选择，例如 `Slim压风散热器Pro`；下方显示 IP 地址输入和保存按钮。
- 用户开启 `虚拟串口/COM 兼容模式`：显示启用设备选择；下方显示端口、波特率、检测端口和保存按钮。
- 用户关闭兼容模式：不删除设备档案，只是不再允许普通连接流程使用对应连接方式。

重要边界：

- 自动扫描只适用于内置设备或已录入了匹配规则的设备。
- 未录入匹配规则的 DIY BLE/HID 设备仍需要先在设备管理里新增或导入。
- 设置页不直接编辑高级协议模板，避免普通连接流程变复杂。
- 首页、设置页、设备管理显示的启用设备名称必须保持一致。

## WiFi 智能启停策略（2.2.1）

Slim压风散热器Pro 当前按只支持风速控制和状态读取处理。2.2.0 只给 WiFi 连接增加 IP 扫描/发现能力，不实现、不显示 WiFi 智能启停。

2.2.1 如要加入 WiFi 智能启停，只能按以下策略：

- 如果设备档案声明原生 start/stop 命令，则走原生命令。
- 如果没有原生命令，只能做软件侧实验功能。
- 软件侧实验功能必须默认关闭。
- 首次启用必须做停机测试。
- 必须有恢复原速度、温度保护、失败回滚、失败自动禁用。
- 设备管理胶囊应区分 `智能启停`、`智能启停·原生`、`智能启停·软件`、`智能启停·实验`。

### Slim压风散热器Pro 的难点

当前 WiFi 设备的大风险不是“显示一个开关”，而是它可能没有固件级智能启停接口，只有百分比风速控制。这样 FanControl 只能尝试通过软件发送 `0%` 或最低停机值来模拟停机。

需要先确认：

- 固件是否接受 `0%`。
- `0%` 是真实停机，还是只是不再降低到更低转速。
- 设备状态接口能否稳定返回当前速度/目标速度/在线状态。
- 发送 `0%` 后，恢复原速度是否可靠。
- 设备断网、断电、延迟、接口失败时，是否会导致长期停机或无法恢复。

### 软件侧智能启停状态机

如果没有原生命令，软件侧智能启停必须做成独立状态机，而不是简单地在温度低时发 `0%`。

建议状态：

- `disabled`：未启用。
- `probing`：测试当前设备是否支持停机值。
- `running`：正常运行，跟随手动/曲线/自动控制。
- `stopping`：正在下发停机值并等待确认。
- `stopped`：已由软件侧停机。
- `starting`：正在恢复到安全速度或原控制模式。
- `recovering`：探测失败、状态读取失败或温度保护触发后的恢复。
- `failed`：多次失败，自动禁用该功能。

### 安全规则

- 默认关闭，并标注为实验功能。
- 首次启用必须执行“测试停机并恢复”。
- 停机前保存原控制模式、目标速度、曲线/手动状态。
- 停机后必须读取状态确认；无法确认时立即恢复。
- 温度超过启动阈值或安全阈值时立即恢复。
- 设置最小停机时间、最小运行时间和温度迟滞，避免频繁启停。
- 发送失败、状态读取失败、恢复失败都要进入保护逻辑。
- 连续失败达到阈值后自动禁用，并在 UI 提示。
- 应保留“紧急恢复风速”入口。

### 与参考 HID 智能启停的区别

- 飞智 HID 智能启停是固件命令 `0x0D`，设备自己执行。
- WiFi 软件侧智能启停是 FanControl 在运行时模拟，不能假装成固件原生能力。
- 如果 Windows 已经完全关机，软件侧逻辑无法继续延迟执行；只能在关机/退出前尽量发送即时停机或恢复命令。
- 因此 WiFi 智能启停 UI 应显示为 `智能启停·软件·实验`，直到固件提供原生命令。

### 推荐拆分

- 2.2.0：只保留 Slim压风散热器Pro 的风速控制和状态读取，增加 IP 扫描/发现，不开放 WiFi 智能启停。
- 2.2.1：实现 WiFi 停机值探测、恢复测试和状态机。
- 2.2.1 或后续小版本：在真实/模拟 WiFi 设备上验证可靠后，再允许用户启用。
